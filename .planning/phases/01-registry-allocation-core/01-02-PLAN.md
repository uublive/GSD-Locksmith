  <action>
    Implement the complete collision detection and retry loop in hooks/claim-number.sh per D-09.

    The collision flow must follow this exact sequence:
    1. Build UPDATED_REGISTRY with new claim(s) appended.
    2. If GSD_DRY_RUN is set, skip to dry-run output and exit 0 (no write, no collision check).
    3. Call write_registry "$UPDATED_REGISTRY". Capture the exit code. If non-zero, print "ERROR: gist write failed" to stderr and exit 2.
    4. Re-read immediately: REGISTRY_AFTER="$(read_registry)". This re-read serves two purposes: collision detection AND write confirmation (Pitfall 4).
    5. Run collision check using jq: `echo "$REGISTRY_AFTER" | jq --argjson n "$NEXT_NUM" --arg t "$TYPE" --arg o "$owner" '[.claims[] | select(.type==$t and .number==$n and .owner!=$o)] | length > 0'`. Capture result as a string ("true" or "false").
    6. If collision is "false": success path — print the success summary to stdout and exit 0.
    7. If collision is "true" (first collision): print near-miss warning to stderr: "WARNING: Collision detected on $TYPE $NEXT_NUM — another developer claimed it first. Retrying...". Extract the competing owner's name from REGISTRY_AFTER using `jq -r --argjson n "$NEXT_NUM" --arg t "$TYPE" '[.claims[] | select(.type==$t and .number==$n)] | .[0].owner // "unknown"'`. Print "Competing owner: $COMPETING_OWNER" to stderr.
    8. Retry: recompute NEXT_NUM from REGISTRY_AFTER (the fresh data), rebuild UPDATED_REGISTRY with the new number, call write_registry again.
    9. Re-read again: REGISTRY_AFTER2="$(read_registry)".
    10. Run collision check again on REGISTRY_AFTER2 with the new NEXT_NUM. If collision is "false": success — print success summary with a note "(after retry due to collision)" and exit 0. If collision is "true" again: print error to stderr "ERROR: Collision persists after retry on $TYPE $NEXT_NUM. Another developer ($COMPETING_OWNER2) claimed the same number. Resolve manually by editing the registry or running this command again." Exit 2.

    The retry logic must NOT recurse — it is a flat two-attempt structure (one write + one retry write = two writes maximum).

    All error and warning messages go to stderr. Only the final success summary goes to stdout.
  </action>
