- git hook fires in post-merge context: no staged files, HEAD is already on the merged state

git post-merge hook specifics:
- Receives one argument: $1 = 1 if squash merge, 0 if regular merge
- HEAD at time of hook = the result commit (merge commit or fast-forward target)
- The branch that was merged IN is NOT directly available from git vars; use MERGE_HEAD (set during merge, unavailable in post-merge) — instead, read the merged branch name from ORIG_HEAD diff or from $GIT_REFLOG_ACTION env
- Simplest approach: use `git reflog -1` or parse `git log --merges -1 --pretty=%s` — but these are unreliable
- Most reliable approach: read the merged branch from the reflog message: `git reflog show --format="%gs" HEAD -1` outputs "merge <branch>: Fast-forward" or "merge <branch>: Merge made by..."
- Extract branch name with: `git reflog show --format="%gs" HEAD -1 | sed 's/^merge \(.*\): .*/\1/'`
- If branch extraction fails (e.g., rebase workflow), fall back to: do nothing and exit 0 cleanly

Create `.githooks/post-merge` following the exact same structural conventions as `.githooks/pre-merge-commit`.

Script structure:

1. Shebang + header comment block (same format as pre-merge-commit): phase, purpose, limitation note (post-merge fires after all merge types including fast-forward but NOT after rebase), activation command.

2. `set -euo pipefail` and `REPO_ROOT="$(git rev-parse --show-toplevel)"`.

3. Source both libraries:
   ```
   source "$REPO_ROOT/hooks/lib/common.sh"
   source "$REPO_ROOT/hooks/lib/gist.sh"
   ```

4. Run `check_deps` and `load_config` (same pattern as claim-number.sh) — but wrap in `|| exit 0` so a missing gist config does NOT block the merge. If deps or config fail, print a warning to stderr and exit 0 (post-merge should never block the user — the merge already happened).

5. Determine the merged branch name. Use git reflog to extract it:
   ```bash
   MERGED_BRANCH="$(git reflog show --format="%gs" HEAD -1 | sed 's/^merge \(.*\): .*/\1/' || true)"
   ```
   If `MERGED_BRANCH` is empty or equals the full reflog line unchanged (sed didn't match), exit 0 with a verbose log: "post-merge: could not determine merged branch name, skipping stale release."

6. Read the registry:
   ```bash
   REGISTRY="$(read_registry 2>/dev/null)" || { verbose_log "post-merge: registry read failed, skipping"; exit 0; }
   ```
   If the result is not valid JSON (pipe through `jq -e .`), exit 0 with verbose log.

7. Check if any active claims match the merged branch:
   ```bash
   MATCH_COUNT=$(echo "$REGISTRY" | jq --arg b "$MERGED_BRANCH" '[.claims[] | select(.branch==$b and .status=="active")] | length')
   ```
   If `MATCH_COUNT` is 0, print "post-merge: no active claims for branch $MERGED_BRANCH — nothing to release" and exit 0.

8. Build updated registry with matching claims set to `"status":"released"`:
   ```bash
   UPDATED="$(echo "$REGISTRY" | jq --arg b "$MERGED_BRANCH" '.claims |= map(if .branch==$b and .status=="active" then .status="released" else . end)')"
   ```

9. Write updated registry. Wrap write_registry call so failure exits 0 (non-blocking):
   ```bash
   write_registry "$UPDATED" || { echo "WARNING: post-merge registry write failed — stale claims for $MERGED_BRANCH not released" >&2; exit 0; }
   ```

10. Print success: `echo "GSD: Released $MATCH_COUNT claim(s) for merged branch $MERGED_BRANCH"` to stdout.

11. `chmod 750 .githooks/post-merge` after creation.

Critical constraints:
- This hook MUST NEVER block a merge — all error paths exit 0 (not exit 1).
- The merged branch extraction must handle both "merge feature/foo: Fast-forward" and "merge origin/feature/foo: Merge made by the 'ort' strategy." formats.
- Do NOT use set +e anywhere — use `|| exit 0` or `|| true` wrappers instead (same pattern as verbose_log || true in common.sh).
- GSD_DRY_RUN=1 should be respected: skip write_registry but still print what would be released. Check `[[ -n "${GSD_DRY_RUN:-}" ]]` before writing.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| post-merge hook → gist registry | Hook writes to shared gist after every merge; malformed write could corrupt registry for all team members |
| merged branch name extraction | Reflog parsing could produce wrong branch name if git version or workflow differs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04 | Tampering | post-merge → write_registry | mitigate | Validate registry JSON with jq -e before writing; write failure exits 0 with warning (non-blocking) |
| T-04-05 | Denial of Service | post-merge hook failure blocking developer workflow | mitigate | ALL error paths exit 0 — hook failure never blocks post-merge workflow; only writes warning to stderr |
| T-04-06 | Information Disclosure | branch name in reflog extraction | accept | Branch names are already in gist registry (not new exposure); team of 3, internal repo |
| T-04-07 | Elevation of Privilege | incorrect branch match releases wrong claims | mitigate | Branch match is exact string equality (.branch==$b); no wildcard/prefix matching |
