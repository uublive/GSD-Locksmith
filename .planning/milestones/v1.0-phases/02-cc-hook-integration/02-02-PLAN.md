**Test 1 — Milestone hook fires and claims number:**
Type /gsd-new-milestone in the CC chat input and press Enter.
Expected:
- Before the command executes, Claude mentions a milestone number was claimed
- The additionalContext is visible in Claude's response or Claude uses the claimed number explicitly
- Running `GSD_DRY_RUN=1 bash hooks/claim-number.sh milestone` from terminal shows the number in the registry OR the CC response references the claimed number

**Test 2 — Phase hook fires and claims number:**
Type /gsd-new-phase 1 in the CC chat input and press Enter.

- Before the command executes, Claude mentions a phase number was claimed under milestone 1
- Claude uses the correct phase number in its response

**Test 3 — Hook blocks on failure (HOOK-03 critical):**

Restore the real gist_id in .claude/gsd-team.json after this test.

**Test 4 — Clean stdout, no contamination (HOOK-04):**
Review the CC session output from Tests 1 and 2. If the hook JSON was correctly emitted:
- CC injects the additionalContext text into Claude's prompt (this is the expected behavior)
- There is no garbled output, no shell welcome banners, no warning about invalid JSON
- Claude responds coherently using the claimed number

If Test 3 shows the command proceeding despite the bad gist_id, the hook is using exit 1 instead of exit 2 — go back to plan 02-01 and fix all exit code paths in the wrapper.

If auth fails in the CC session (Pitfall 3 — gh auth fails in CC subprocess):
- Run `gh auth status` from terminal to confirm terminal auth works
- If terminal works but CC fails: set GH_TOKEN environment variable in your shell profile BEFORE launching CC (note: wrap in [[ $- == *i* ]] guard per HOOK-04)
