  <action>
Create `scripts/install-hooks.sh` as an idempotent installer. The script must:

1. Use `set -euo pipefail` and `REPO_ROOT=$(git rev-parse --show-toplevel)`.

2. Validate prerequisites before doing anything: check that `jq` is installed (exit 1 with install hint if not), check that `git` is installed (always present, but check anyway), check that `.claude/gsd-team.json` exists with a non-empty `gist_id` field — if missing, print a clear error directing the user to create the gist first (see README-HOOKS.md for the step), then exit 1.

3. Configure git hooks via `git config core.hooksPath .githooks`. This is idempotent — git config always overwrites with the same value. Print "git hooks configured: core.hooksPath = .githooks" to stdout on success.

4. Merge CC hook entry into `.claude/settings.json`. The merge logic must be idempotent:
   - Read current `.claude/settings.json` with `jq`.
   - Check whether `.hooks.PreToolUse[]` already contains an entry with `command` matching `*cc-pretool-claim.sh` using `jq` select.
   - If the entry already exists, print "CC hook already configured in .claude/settings.json — skipping" and skip the write.
   - If the entry does not exist, use `jq` to add the entry to `.hooks.PreToolUse` array. The entry structure must match exactly:
     ```
     {"matcher":"Bash","hooks":[{"type":"command","command":"${CLAUDE_PROJECT_DIR}/hooks/cc-pretool-claim.sh"}]}
     ```
   - Write the merged JSON back to `.claude/settings.json`. Use a tmpfile approach: write to a temp file, then `mv` it to replace the original (atomic write).
   - Print "CC hook configured in .claude/settings.json" on success.

5. Print a final summary: "GSD hooks installed. Run: bash tests/test-validate.sh to verify validation."

6. `chmod 750 scripts/install-hooks.sh` after creation — consistent with Phase 2/3 hook convention.

Important constraints:
- Do NOT use `echo` with JSON — always use `printf '%s'` or `jq -n` to avoid flag interpretation issues (per [01-01] decision in STATE.md).
- The `.claude/settings.json` write must preserve ALL existing top-level keys — do not blow away the current content.
- `${CLAUDE_PROJECT_DIR}` in the command string must remain a literal shell variable reference in JSON — do not expand it during install. Use single quotes or `--arg` to keep it literal.
- If `.claude/settings.json` does not exist, create it from scratch with the minimal structure containing only the hook entry.
  </action>

|----------|-------------|
| install script → settings.json | Script writes JSON to .claude/settings.json; malformed jq output could corrupt the file |
| install script → git config | Sets core.hooksPath; benign (no secret data) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | scripts/install-hooks.sh → .claude/settings.json | mitigate | Atomic write via tmpfile + mv; validate JSON with jq -e before mv to prevent writing corrupt content |
| T-04-02 | Spoofing | gist_id in .claude/gsd-team.json | accept | Team of 3, file is committed to repo — substitution would require commit access; out of threat scope |
| T-04-03 | Denial of Service | install script exits non-zero blocking setup | mitigate | Clear error messages with fix commands so developers are never stuck silently |
</threat_model>
